Abstract
The widespread adoption of large language models (LLMs)
has made GPU-accelerated inference a critical part of modern
computing infrastructure. Production inference systems rely
on CUDA kernels to implement core transformer operations,
yet these kernels are highly susceptible to memory-safety
bugs due to model-dependent tensor layouts, intricate memory
indexing, and massive thread-level parallelism. Such bugs
can corrupt model weights, crash inference services, or even
enable adversarial attacks. Existing techniques either depend
on unavailable hardware, incur high overhead, or fail to handle
kernel inputs with variable lengths, and none can effectively
detect CUDA memory bugs in LLM inference systems.
This paper presents Model2Kernel, the first practical sys-
tem for automatically verifying the memory safety of CUDA
kernels used in LLM inference. Model2Kernel performs
model-aware dynamic analysis to determine how each model
invokes kernels and to classify kernel arguments as either
fixed by the model architecture or controlled by model users.
Using this information, Model2Kernel then applies CUDA-
specialized symbolic execution, supported by new abstrac-
tions for dynamic tensor memory and thread identifiers, to
accurately pinpoint memory bugs in kernels. In the evaluation
on CUDA kernels and models from vLLM, Hugging Face,
and recent LLM research papers, Model2Kernel discovers 353
previously unknown bugs while producing only nine false
positives, demonstrating its effectiveness.

Introduction
The rapid adoption of large language models (LLMs) has
driven an explosion of inference workloads across chat bots [1,
2,22], search [44,91], recommendation [5,80,86], copilots [15,
40,90], scientific computing [7,45,77], and countless other
areas [8, 10, 84]. These services now operate at large scale
and serve real users, making LLM inference systems a critical
part of modern computing infrastructure.
LLM inference systems rely on a layered software archi-
tecture. Model developers publish Python model files and
weights—such as those released on Hugging Face [29]—
which inference frameworks (e.g., vLLM [42], SGLang [87],
Transformers [30]) load and execute. These frameworks
manage inference requests and register GPU kernels to link
Python-level logic with host-side wrapper code. The wrap-
per code prepares CUDA kernel inputs and configures kernel
launches. At the lowest layer, CUDA kernels perform the
core linear algebra operations required by transformer archi-
tectures on the GPU. This design supports rapid innovation
but substantially increases system complexity.
Memory-safety bugs remain a persistent challenge [72],
and CUDA kernels are particularly vulnerable for three main
reasons. First, fine-grained index calculations across large
tensors and thousands of threads make kernels prone to out-
of-bounds accesses and data races. Second, as model archi-
tectures grow larger [84] and inference sequences become
longer [23, 74], the risk of integer overflows in CUDA ker-
nels increases. Third, the complexity of the entire inference
stack can produce tensor layouts that violate kernel assump-
tions [16, 52], further increasing the potential for memory-
safety errors.
Memory-safety bugs in CUDA kernels can lead to severe
consequences. Out-of-bounds accesses and data races often
trigger “illegal memory access” errors on the GPU, poten-
tially crashing the entire inference service [63]. Recent work
has also shown that such vulnerabilities can be exploited to
modify arbitrary GPU memory or even achieve arbitrary code
execution [25, 34, 57]. Because inference systems interact
with external users and may receive attacker-controlled inputs,
exploitable memory bugs in kernels pose a serious security
risk, potentially allowing adversaries to alter deployed model
weights or compromise the entire system.
Existing techniques for detecting memory bugs in CUDA
kernels fall into two main categories. Dynamic methods
track metadata for GPU memory buffers and check whether
pointer values fall within valid bounds before dereferenc-
ing [28,43,43,49,59,60,73,82,88,89,92]. However, these
methods either incur high runtime overhead or rely on ex-
tra hardware not yet widely available. Static tools, on the

other hand, cannot handle buffers with sizes determined

We argue that symbolic execution provides a promising
approach for detecting memory-safety issues in CUDA ker-
nels. Many critical vulnerabilities (e.g., integer overflows,
buffer overflows, data races) can be formulated as constraint-
satisfaction problems over indices, tensor shapes, and thread
identifiers, a domain where symbolic execution excels. In
addition, CUDA kernels are generally simpler than typical
C/C++ programs: they primarily operate over multidimen-
sional tensors, use few custom data structures, and rarely
involve complex pointer operations (e.g., pointers to pointers)
or indirect function calls. These characteristics make CUDA
kernels easy for symbolic analysis.
However, applying symbolic execution to CUDA kernels
presents several challenges. First, existing symbolic execution
techniques have difficulty handling dynamic arrays, yet most
CUDA kernels in LLM systems operate on tensors whose
shapes are often determined only at runtime. Second, CUDA
kernels execute across thousands of parallel threads, and scal-
ing symbolic execution to these threads is non-trivial. Third,
kernel behavior may vary across different model architec-
tures, and automatically inferring these architectures and un-
derstanding how they affect kernel execution remain complex
due to the intricacies of the full inference stack.
To address these challenges, we design Model2Kernel,
a fully automated analysis framework that verifies whether
CUDA kernels exhibit memory-safety issues when used to-
gether with a given LLM model. Model2Kernel targets four
types of bugs in CUDA kernels: buffer overflows, integer
overflows, data races, and NULL pointer dereferences, and
supports models sharing the same format as those released on
Hugging Face. As shown by Figure 1, Model2Kernel consists
of two major components: cuKLEE and HFProbe.
cuKLEE is a symbolic execution engine specialized for
CUDA kernels. To handle input tensors whose memory lay-
outs are determined at runtime, we model each tensor as a
distinct and widely separated memory region. This abstrac-
tion simplifies pointer–tensor association and makes bounds
checking more tractable. We then introduce symbolic vari-
ables to represent each tensor’s base address, element count,
dimensions, and other properties. With these variables, cuK-
LEE can handle more than 100 tensor methods by adding the
appropriate constraints to its execution states. We also model
CUDA thread identifiers as symbolic variables, enabling uni-
fied symbolic execution that captures both thread-shared and
thread-specific computations in a single pass. This design
allows cuKLEE to scale to thousands of CUDA threads.
HFProbe “executes” each model without requiring GPU
hardware. It profiles how the model invokes CUDA kernels
and automatically identifies which kernel inputs are deter-
mined by the model architecture (and therefore should re-
main fixed during cuKLEE’s analysis) and which inputs are
controlled by end users (and should be treated as variable).
Incorporating this information allows cuKLEE to analyze
kernels more efficiently and accurately. In addition, HFProbe
statically identifies all CUDA kernels that may be invoked by
a given model and automatically mutates the model’s configu-
rations to trigger more kernels, ensuring more comprehensive
analysis of both the model and its CUDA kernels.
We evaluate Model2Kernel on CUDA kernels and models
from three sources: all text-generation models in vLLM [42]
and the related kernels, all Hugging Face models with custom
kernels, and models and kernels from four recent research
publications [39,55,64,81]. In total, Model2Kernel identifies
353 memory bugs, including 328 integer overflows and 25
out-of-bounds accesses, with only nine false positives, demon-
strating its effectiveness in detecting memory safety issues
for CUDA kernels. We further compare Model2Kernel with
Honeycomb [54], GKLEE [47], and ESBMC-GPU [65] on
20 known vLLM CUDA kernel bugs. Model2Kernel success-
fully detects 15 of these bugs, far more than the baseline
techniques, highlighting its good bug coverage and clear im-
provement over existing techniques. An ablation study shows
that both HFProbe and cuKLEE are critical for Model2Kernel
to achieve this level of effectiveness.
In sum, we make the following contributions:
• We design cuKLEE, a CUDA-specific symbolic execu-
tion engine that supports tensors with dynamic shapes
and scales to thousands of CUDA threads.
• We develop HFProbe, a dynamic model profiler that an-
alyzes how models invoke CUDA kernels and infers
model-dependent kernel arguments.
• By integrating cuKLEE with HFProbe, we build
Model2Kernel, a fully automated framework for detect-
ing CUDA memory bugs in LLM inference systems, and
we demonstrate its effectiveness through comprehensive
experiments.

2.1
Hugging Face Models
Hugging Face is the largest platform for sharing pre-trained
models [29], offering both a model marketplace and libraries
that facilitate model integration into LLM inference frame-
works (e.g., vLLM [42], SGLang [87], Transformers [30]). To-
day, it hosts millions of models, including major open-source
foundation models (e.g., Qwen, LLaMA) and their variants,
making it a central hub for the AI community.
A model released on Hugging Face typically contains four
types of files. The config.json file defines the model’s architec-
ture, including layers, hidden size, activation functions, and
other structural parameters. The model-weight files store the
trained parameters. The Python code files define classes that
extend nn.Module and implement the model’s architecture.
One top-level class assembles all model components together,
with its forward() method as the entrance of the model’s
whole forward pass. For example, Figure 2a shows simplified
model code for DeepseekV3ForCausalLM, and forward()
in line 4 begins the model’s inference process. Finally, doc-
umentation files (e.g., README.md) provide instructions,
usage examples, and licensing information.
2.2
The CUDA Programming Language
CUDA is the dominant general-purpose programming lan-
guage for exploiting GPU computational power [21], and
most GPU kernels in inference systems are written in CUDA.
CUDA source files (.cu files) typically contain two types of
code: host wrapper functions and GPU kernel functions. Host
wrappers run on the CPU and are exposed to Python code in
inference systems via registration or binding macros [36,67]
(e.g., Figure 2b). They process input Tensors from Python
code into kernel arguments, configure launch parameters (e.g.,
3
number of thread blocks, threads per block, shared-memory
size), and call the kernels. CUDA kernels run on the GPU.
They define the computation performed by each parallel
thread. Under CUDA’s execution model, a kernel runs across
many threads organized into blocks and grids. Each thread
uses built-in identifiers (e.g., blockIdx.x, threadIdx.x) to
determine its location in this hierarchy and the portion of
input data it processes.
Function fused_add_rms_norm() in Figure 2c is a wrap-
per function. It computes the model’s hidden size and
the number of input tokens in lines 2–3, sets the num-
ber of blocks to the token count (line 4) and threads per
block to 256 (line 5), and then launches kernel function
fused_add_rms_norm_kernel() with pointers to the under-
lying memory of the input tensors, the epsilon parameter,
and the computed hidden size and token number in lines 8–9.
Figure 2d shows the launched CUDA kernel. It fuses
element-wise addition, RMS normalization, and scaling by
weight. To improve performance, the kernel uses vectoriza-
tion. It calculates how many vector units fit into the hidden
size in line 10, allocates a shared-memory variable that all
threads in the block can access in line 11, and casts the input
pointers to vectorized types in lines 14–16. The loop in lines
18–25 distributes work across threads. Each thread determines
which part of data it should process based on the block’s start-
ing offset (blockIdx.x * vec_hidden_size), its thread ID
(threadIdx.x), and additional loop increments equal to the
block size (blockDim.x). In each iteration, a thread loads
a vector in line 21, adds residuals in line 22, and accumu-
lates the squared sum into the thread-local variance in line
23. Next, a block-wise reduction sums variances from all
threads in lines 27–29. The kernel computes the RMS nor-
malization coefficient and stores it to the shared memory in
line 31. Finally, the second loop in lines 36–43 uses the same
indexing pattern to apply the normalization, multiplies by
weight, and writes the results back to input.
2.3
Memory Safety Issues in CUDA Kernels
Unfortunately, CUDA kernels in LLM inference systems are
prone to memory bugs (e.g., integer and buffer overflows).
This arises partly because kernels must implement complex,
highly optimized matrix operations to exploit hardware fea-
tures such as caches and vectorized instructions (e.g., line
21 in Figure 2d), increasing the likelihood of programming
errors. Moreover, inference systems integrate kernels and
models from different groups, so a kernel correct for one
model may fail for another. For example, the model’s hidden
size of 8192 in Figure 2a and line 8 of Figure 2c yield width
= 8 and vec_hidden_size = 1024 in Figure 2d. The ker-
nel launches one block per token, so blockIdx.x equals the
number of tokens or the batch size multiplied by the sequence
length. With batch size up to 64 and sequence length up to
128,000, the product blockIdx.x * vec_hidden_size in
line 20 can easily exceed the int32 limit, triggering an inte-
ger overflow. The overflowed value is then used as an index
into input_v, causing a buffer overflow.
We conduct an empirical study to assess the prevalence
and characteristics of CUDA memory bugs in LLM inference
systems. By examining vLLM’s change logs, we identify 20
resolved memory bugs in CUDA kernels, including ten integer
overflows (also causing buffer overflows), eight additional
buffer overflows and two null-pointer dereferences. 17 occur
on input tensors residing on GPU global memory, while three
happen on GPU shared memory. 13 were reported by users
during production, six detected by unit tests, and one found
via code review. These results highlight the prevalence of
CUDA memory errors in LLM inference systems and their
strong association with tensors.
CUDA memory errors can easily lead to incorrect compu-
tations in LLM inference systems. Moreover, GPUs trigger
an “invalid memory access” when unallocated memory is
accessed, which can crash the entire inference system, as
observed in our study [13]. Compounding the risk, recent
research shows that such memory errors can be exploited for
attacks, including arbitrary memory writes and code execution
on GPU [25,34,57], highlighting the urgent need to address
CUDA memory bugs in LLM inference systems.
2.4
Symbolic Execution Techniques
Symbolic execution [79] is a static analysis technique that “ex-
ecutes” programs with symbolic, rather than concrete, inputs.
By systematically exploring execution paths and invoking con-
straint solvers to assess path feasibility, it can generate inputs
that expose potential defects. Numerous symbolic execution
techniques have been developed. They have successfully de-
tected diverse issues (e.g., buffer overflows, integer overflows)
in critical software systems [9,11,17,33,76,78].
Existing symbolic execution techniques struggle with dy-
namic arrays whose sizes are only known at runtime. Con-
sider how KLEE [11] handles the allocation “buffer =
(int *)malloc(n * sizeof(int))” where n is fully user-
controlled and marked symbolic. KLEE forks three states
upon encountering this statement. The first assumes n = 1,
typically producing false positives for later accesses because
real inputs are usually much larger. The second assumes
n > 231, causing KLEE to report “null page access” once
buffer is used. The third treats n as a symbolic uint32, is-
sues a “concretized symbolic size” warning, and terminates
the exploration for the state. KLEE and most symbolic ex-
ecutors handle dynamic arrays in this simplified way because
they use a contiguous memory model. If an array’s size is
symbolic, the addresses of all subsequent buffers also become
symbolic, dramatically complicating memory–access reason-
ing for those buffers.
This limitation is particularly problematic for CUDA ker-
nels, especially those used in LLM inference, because they
4
operate directly on tensor memory whose shapes depend on
model architectures or even on user-provided token counts
(e.g., the dimensions of the input tensor in Figure 2c). Conse-
quently, although prior efforts have explored symbolic execu-
tion for CUDA kernels [6,35,46,48], none can be effectively
applied to CUDA code in modern LLM inference systems.

Overview of Model2Kernel
Model2Kernel is designed to detect memory bugs in CUDA
kernels for LLM inference systems. As shown in Figure 1,
it consists of two components: HFProbe, which collects
model-architecture information, and cuKLEE, which iden-
tifies CUDA memory bugs using this information.
HFProbe supports two backends, vLLM [42] and Trans-
formers [30], enabling it to analyze any model compati-
ble with these frameworks, including all models from Hug-
ging Face [29]. For each CUDA kernel, HFProbe determines
which kernel inputs are model-dependent and which are user-
controlled. For example, when analyzing the model in Fig-
ure 2a, HFProbe infers that the last dimensions of tensors
input, residual, and weight in Figure 2c are all 7168, the
model’s hidden size, while the first dimensions of input and
residual correspond to the number of input tokens.
cuKLEE is a symbolic execution engine specialized for
CUDA that detects out-of-bounds accesses, integer overflows,
data races, and null-pointer dereferences. Using model in-
formation provided by HFProbe, it identifies buggy lines
in CUDA kernels and generates concrete values for user-
controlled kernel inputs that can trigger the bugs. For exam-
ple, for Figure 2c, it reports that when the first dimension of
input is 2,432,000 (batch size 19, sequence length 128,000),
line 20 experiences an integer overflow and line 21 performs
an out-of-bounds access in Figure 2d.
Model2Kernel enhances the safety and reliability of infer-
ence systems in two ways. First, model developers can use it
to verify existing CUDA kernels correctly support newly de-
veloped model architectures and expected input token counts.
Second, CUDA developers can use it to ensure that new ker-
nel optimizations do not introduce memory bugs and remain
compatible with models already deployed in production.

4.1.1
Memory Model of cuKLEE
Tensor Memory. GPUs have three types of memory: global,
shared, and local. All input tensors and tensors created by
wrapper functions reside in global memory and are accessible
to all GPU threads. We model the memory regions of these
tensors as discrete and well-separated, because all input ten-
sors are allocated by the inference system, which prevents
CUDA kernel programmers from making any assumptions
about their layout. Consequently, in CUDA kernels, a pointer
to an element of one tensor never refers to an element of
another tensor through arithmetic computations.
As shown in Table 1, we associate each tensor with a set of
symbolic constraint variables. cuKLEE uses these variables
to construct constraints when it analyzes specific instructions
or tensor methods. Note that the number of dimensions (D)
may remain unknown even at the end of analysis. Therefore,
we maintain a map to record all known dimensions, where
the keys are dimension indices and the values are the corre-
sponding symbolic variables (e.g., d1, d2). For example, line
2 of Figure 2c accesses the last dimension of tensor input
by calling input.size(-1). cuKLEE processes this line by
adding an entry to the map with key −1 and value dD−1, and
it also adds a constraint representing hidden_size is equal
to dD−1 into its execution state.

be allocated dynamically at kernel launch, where cuKLEE

Figure 2d), in which case the size is known directly. cuKLEE
also treats global, shared, and local memory as separate re-
gions and resolves pointer targets only within the region to
which they belong.

4.1.3
Handling CUDA Threads
CUDA organizes threads into blocks and blocks into grids. Al-
though all threads execute the same kernel code, their behavior
differs based on their thread and block identifiers, which pro-
grammers also use to partition input data. cuKLEE follows
this execution model by treating thread and block identifiers
as symbolic variables rather than enumerating all possible
values, allowing a single symbolic execution to represent the
behavior of all threads. Thread-independent bugs are handled
as in sequential programs, while thread-dependent bugs (e.g.,
line 20 in Figure 2d) are captured through constraints over
these symbolic identifiers. cuKLEE then checks the feasibility
of these conditions under the kernel’s launch configuration.
To support this analysis, we introduce two classes of con-
straint variables: six ID variables and six launching vari-
ables, as shown in Table 3. ID variables are constrained
by launching variables according to CUDA semantics (e.g.,
blockIdx.x < gridDim.x). Constraints on launching vari-
ables are obtained when analyzing the kernel launch (e.g.,
blockDim.x=num_tokens in Figure 2c).
For loops that distribute work across threads (e.g., the high-
lighted loops in Figure 2d), cuKLEE analyzes the loop body
only once rather than forking execution at each termination
check. cuKLEE adds the loop’s exit condition (e.g., idx >=
vec_hidden_size) for the remaining execution. The loops’
termination conditions are usually simple to ensure correct-
ness for thousands of threads, making single-pass analysis
enough. cuKLEE also detects data races within such loops by
creating a second set of ID variables that represent a different
thread with the same launch constraints. cuKLEE adds con-
straints indicating the two threads are different by negating
the constraints that the two threads’ six ID variables are equal
pairwise. For each global memory access, cuKLEE checks
whether this second thread may perform a conflicting write
to the same address. A similar check is performed for shared
memory, excluding constraints on blockIdx.x/y/z, since
only threads in the same block can access shared memory.
4.1.4
Additional Considerations
Initial Constraints. Before analyzing each wrapper function,
cuKLEE introduces three constraint variables: BS for batch
6
size, SL for sequence length, and TC for the total number of
tokens, with the relation TC = BS×SL. In addition to the con-
straints provided by HFProbe, we further restrict BS ≤1000
and SL ≤1,000,000. Many recent models already support
context lengths of one million tokens [18,19], and maximum
context lengths have increased by roughly a factor of 30 per
year since 2023 [32]. These bounds therefore reflect both
current and near-future LLM capabilities.
Buggy Constraints. cuKLEE detects four types of bugs.
Data-race detection is described in Section 4.1.3; here, we fo-
cus on the other three. To detect out-of-bounds accesses, cuK-
LEE adds constraints for each memory access via a pointer
indicating that the pointer is either smaller than the base ad-
dress or larger than the upper bound of the buffer. For integer
overflows, it adds a constraint for each addition or multipli-
cation on 32-bit integers specifying the result exceeds the
type’s maximum value. For null-pointer dereferences, cuK-
LEE adds constraints for each pointer dereference indicating
the pointer may have been computed from an unknown base
pointer. We assume that all input tensors are valid and that
their base addresses are properly initialized.
Library and Instruction Support. We leverage API
summarization [38, 53] to handle 91 CUDA APIs (e.g.,
cudaMalloc()) and 39 C/C++ library functions (e.g., meth-
ods of std::vector) that are commonly used by wrap-
per functions. In essence, we use summarization to up-
date the execution state without performing inter-procedural
analysis to explore their internal logic. Similarly, we sup-
port 44 LLVM intrinsics and 59 inline assembly calls
used in CUDA kernels. For example, when processing
llvm.nvvm.read.ptx.sreg.tid.x(), cuKLEE returns the
symbolic variable representing threadIdx.x.
4.2
HFProbe: Dynamic Profiling of HF Models
HFProbe provides analysis contexts for CUDA kernels by
supplying concrete shapes for some (but not all) input tensors
and placing constraints on their values. This reduces the num-
ber of symbolic variables handled by cuKLEE, improving
scalability and decreasing false positives arising from unre-
alistic inputs. As illustrated in Figure 1, HFProbe operates
in four stages. It first performs static analysis to identify all
CUDA kernels each input model may call. It then executes the
model with a real configuration using mocked CUDA kernels
and records how the kernels are actually invoked. Next, it mu-
tates the configuration to exercise additional kernels identified
statically but not triggered dynamically. Finally, it aggregates
the collected information in a post-mortem analysis to derive
analysis contexts for cuKLEE.
4.2.1
Static Call Graph Computation
Given a model’s Python implementation (e.g., Figure 2a), we
build a static call graph using the model’s forward() method
as the root. The analysis is path- and field-sensitive and lever-
ages common coding patterns of Hugging Face models. For
example, we assume that a model object has no aliases within
its own implementation. Our analysis identifies all CUDA ker-
nels used by the model. However, the resulting call graph may
also include spurious kernels that are not actually used. Ex-
isting Python call-graph analysis techniques cannot precisely
analyze Hugging Face models [37,70,83]. For instance, PyCG
does not handle __call__, preventing it from resolving calls
to an nn.Module object—a common practice for invoking a
layer’s forward() method (e.g., line 5 in Figure 2a).
Our analysis operates on the Abstract Syntax Tree (AST) of
the input Python program. It maintains an execution context
where each program object is represented as an instance. The
analysis creates the instance when analyzing the object’s class
constructor. Each instance stores its fields and their values
in a dictionary. The analysis processes each instruction with
respect to this context and also updates it according to the
instruction’s semantics.
Our analysis begins with the model’s __init__() method
to initialize the context. It then analyzes the forward()
method to construct the call graph. For each object field as-
signment (e.g., line 3 in Figure 2a), the analysis updates the
left-hand instance’s dictionary using either the concrete value
or the instance referenced on the right-hand side. For each
function call, it performs interprocedural analysis: it maps the
instances of actual arguments to the formal parameters of the
callee. If the receiver’s type is known only at the base class
level, all methods of possible derived classes are treated as
potential callees. At each branch, the analysis inspects the
context to decide which path is feasible. If both paths may
occur, it clones the context and analyzes each path indepen-
dently. When encountering Python bindings to CUDA kernels
(e.g., line 20 in Figure 2a), it records that the model calls the
corresponding CUDA kernels. All Python bindings and their
links to CUDA kernels (e.g., Figure 2b) are identified in a
separate analysis using regular expressions.

we inform the LLM that the necessary framework configura-

config.json.
4.2.4
Context Generation
Given a configuration, we execute the model with varying
batch sizes and sequence lengths, which together determine
the number of input tokens. For each kernel, we analyze the
recorded values of tensor dimensions, primitive parameter
values, and the minimum and maximum values of integer-
valued tensors. From this data, we infer four categories of
constraints that together form the analysis context of cuKLEE:
1) strict linear relationships with batch size, sequence length,
or token count; 2) invariant values determined by the model
architecture; 3) equality relationships between the values;
and 4) residual parameters that do not fall into the above
categories but may still exhibit indirect dependence on the
model architecture; for these, we impose an upper-bound
constraint using the maximum value observed.
For example, after executing the model in Figure 2a with a
configuration [31], we infer that the last dimensions of input,
residual, and weight in Figure 2c are all 7168, fixed by
the model architecture, that the first dimensions of input and
residual are token count, and that epsilon equals 1e−6.

5.2
Coverage and Advancement
5.2.1
Methodology
We use the 20 known vLLM bugs analyzed in Section 2.3
to evaluate coverage. Most of these bugs were reported by
vLLM users, and the corresponding model information is not
available. However, the bug reports or developer discussions
typically include the kernel in which the bug occurs and the
required inputs (e.g., tensor shapes). Therefore, in this experi-
ment, we evaluate only cuKLEE. We retain the input fields
mandated by the model architecture and treat all other inputs
as symbolic when running cuKLEE.
We compare cuKLEE with three static CUDA analy-
sis tools: Honeycomb’s validator [54], GKLEE [47], and
ESBMC-GPU [65]. These tools cannot run directly on vLLM
kernels due to incompatibilities with modern C++, CUDA, as
well as PyTorch and vLLM CUDA libraries (e.g., Tensor).
Moreover, GKLEE and ESBMC-GPU require a main() as
their entry point, which vLLM kernels lack. To enable com-
10
parison, we simplify the vLLM kernels by rewriting modern
CUDA/C++ features with legacy constructs, removing Py-
Torch and vLLM CUDA dependencies, and adding a main()
function for each kernel that replaces the original wrapper and
allocates GPU memory for tensors. These simplified kernels
preserve the root causes of the bugs. Because Honeycomb
targets HIP kernels [4], we additionally translate CUDA APIs
to HIP equivalents using hipify-perl [3]. We do not compare
with other related tools because they are not free [20,62] or
their code is not available [14,35,46,48].

As shown in Table 5, cuKLEE identifies 15 of the 20 known
bugs, including nine integer overflows, five out-of-bounds
accesses, and one NULL-pointer dereference, demonstrating
strong coverage of real memory errors in CUDA kernels.

causing it to miss one bug that depends on this construct. For
the simplified kernels, cuKLEE still fails to detect the first

5.3
Rationality of Components
5.3.1
Methodology
We conduct an ablation study to assess the contribution of
each component of Model2Kernel. We create several vari-
ants of Model2Kernel by disabling individual components
and compare their effectiveness against the full-featured ver-
sion, using the vLLM models and kernels under the same
experimental setup as Section 5.1.
Specifically: 1⃝Without HFProbe. We run cuKLEE di-
rectly on kernel wrapper functions without providing any
guidance from HFProbe. 2⃝Without cuKLEE. We execute
each model on real GPUs using both original and mutated
configurations, and record runtime errors (e.g., “illegal mem-
ory access”). To parallelize experiments, we use multiple
GPU types (four RTX 5090s and four H100s); since we only
check for error occurrence, hardware differences do not af-
fect the results. 3⃝Without configuration mutation. We run
Model2Kernel on the collected config.json files and vLLM’s
default configurations, without applying any mutations.

Experimental Results
The full-featured version of Model2Kernel detects the most

6
Limitations and Discussion
Model2Kernel currently focuses on Hugging Face models
and CUDA kernels with tensor inputs, which cover a large
portion of real-world usage. However, due to the ecosystem’s
diversity and rapid evolution, Model2Kernel does not yet sup-
port components such as Triton-based kernels or TensorFlow
models. Its core ideas—mutating model configurations to ex-
plore diverse execution paths, modeling dynamic arrays as
discrete memory regions, and symbolizing thread IDs—can
be generalized to other bug types and CUDA kernels. Extend-
ing cuKLEE to NPU instruction sets and NPU kernels is a
promising direction for future work.
In Section 4.2, we mutate execution configurations to trig-
ger additional CUDA kernels. While this increases coverage,
collected contexts still represent only a small fraction of real
LLM inference scenarios. Developing more systematic meth-
ods to explore inference frameworks and validate kernels
under broader, realistic contexts remains future work.
In our experiments, cuKLEE analyzes each kernel for up
to one hour. For certain kernels (e.g., gptq_marlin_gemm(),
moe_wna16_marlin_gemm()), not all paths are explored, po-
tentially missing memory bugs. cuKLEE builds on KLEE and
inherits its path exploration strategies, originally designed for
C/C++. Since CUDA kernels may follow different program-
ming patterns, optimizing path exploration for CUDA is a key
direction to improve cuKLEE’s effectiveness.
7
Related Work
Exploiting CUDA Memory Bugs. Prior work shows that
CUDA memory bugs are highly exploitable [34, 69]. They
can enable control-flow hijacking [25,57], mind-control at-
tacks [63], and side-channel attacks [58,66,85], leading to se-
vere inference degradation or arbitrary code execution. These
findings motivate our systematic analysis of CUDA kernels
to detect memory bugs in them.
Static techniques have been developed for CUDA kernels.
ProofWright [14] uses Vercons to verify kernels against LLM-
generated specifications. COVT [50] translates CUDA into
C/C++ and then detects integer and buffer overflows. Honey-
comb’s validator [54] ensures memory accesses remain within
predefined coarse-grained regions. Tong et al. extend Dartag-
nan with GPU consistency models [75]. Symbolic-execution-
based methods have also been proposed [6, 35, 46–48, 65].
Overall, these static techniques struggle with dynamically
sized memory buffers, do not support the widely used Tensor
library, and cannot handle varied kernel launch configurations.
We evaluate three of them [47,54,65] in Section 5.2 and find
they are limited in usability and less effective than cuKLEE.
Commercial static tools also exist [20,62], but their technical
details are unavailable for analysis.
Dynamic Techniques. NVIDIA provides several mech-
anisms to enforce GPU memory safety. By default, the
GPU MMU reports “illegal memory access” errors when
untracked virtual addresses are accessed [56]. CUDA-
MEMCHECK [59] and its successor Compute Sanitizer [60]
record metadata for GPU buffers, enabling detection of a
broader range of memory errors at the cost of increased run-
time overhead. Researchers have also devoted efforts in this
area. GPUShield [43], GPUArmor [92], cuCatch [73], and
LAK [82] build on similar principles as Compute Sanitizer,
with the first three emphasizing lower runtime overhead and
latter using lock-key metadata to provide stronger memory
safety. GMOD [26, 27] pads memory buffers and detects
overflows by monitoring changes to the padding. IMT [71]
leverages ECC to ensure GPU memory safety. Race detec-
tion tools, including BARRACUDA [28], GRace [88], GM-
Race [89], and LDetector [49], track the last-accessing thread
for each memory location and identify unsynchronized con-
current accesses. Despite these advances, these techniques
rely on specific inputs to trigger bugs, require instrumentation
or hardware support, and many incur large runtime overhead.
8
Conclusion
This paper presents Model2Kernel, an automated tool that
combines dynamic model profiling with CUDA-specialized
symbolic execution to detect CUDA memory bugs in LLM
inference systems. Model2Kernel identifies more than 300
CUDA memory bugs in popular LLM inference frameworks
when they perform inference with real models, demonstrating
its practical effectiveness. Future work can further improve
the efficiency of symbolic execution for CUDA kernels and
extend Model2Kernel to support NPU kernels.
